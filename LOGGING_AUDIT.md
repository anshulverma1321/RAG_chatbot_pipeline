# Logging & Observability Audit Report

This document reviews the existing logging implementation for the Multimodal RAG platform, identifies gaps in current logging coverage, and maps out requirements for the structured logging upgrade.

---

## 1. Current Logging Implementation

The platform's logging is configured and managed centrally in [logger.py](file:///d:/RAG_tool-zip/RAG_tool/RAG/logger.py). 

### Current Log Files
* **`logs/app.log`**: Consolidated application log containing start/success ingestion details, queries, context summaries, and errors.
* **`logs/errors.log`**: Dedicated log for caught exceptions and error messages across all modules.
* **`logs/ingestion.log`**: Captures start and success of document ingestion processes.
* **`logs/retrieval.log`**: Stores document queries, vector retrieval lists, and similarity scores.
* **`logs/chat.log`**: High-level conversational logs including user queries, LLM context references, and model response times.

### Current Logger Configuration
* Loggers are configured manually using `logging.FileHandler` with UTF-8 encoding.
* Standard message format is raw text (`handler.setFormatter(logging.Formatter('%(message)s'))`).
* Log propagation is disabled (`logger.propagate = False`) for component-specific loggers.
* Custom multiline strings with block separators (`===`, `---`) are compiled manually inside logger helper functions (`log_ingestion_start`, `log_ingestion_success`, etc.).

## 2. Identified Gaps and Gaps in Observability

### A. Non-Structured Logging Format
* **Problem**: Logs are formatted as unstructured, human-readable multiline text blocks.
* **Impact**: Programmatic log analysis, parsing, searching, or ingestion into external monitoring solutions (ELK, Datadog) is highly complex and error-prone.

### B. Lack of Request Tracing (`request_id`)
* **Problem**: There is no tracing identifier propagated across request lifecycles.
* **Impact**: It is impossible to group log statements generated by a single transaction, especially for asynchronous processes like `/upload` where ingestion runs in a background thread distinct from the API request thread.

### C. Missing Stage-Level Observability
* **Problem**: Current logs focus on entry/exit points (e.g. Ingestion Start / Ingestion Success) rather than internal pipeline stages.
* **Impact**: In the event of a failure, it is difficult to determine which specific sub-module (e.g., dynamic VLM routing, PDF page parsing, table extraction, text normalization, SQLite inserting, or Qdrant vector upserting) failed.

### D. No Log Rotation
* **Problem**: Logs are written to files using standard `FileHandler`.
* **Impact**: Under production loads, log files will grow indefinitely, risking disk exhaustion.

### E. Unstructured Access and Performance Timings
* **Problem**: FastAPI access logs are managed by standard Uvicorn output. Timings are written as plain-text lines to custom logs.
* **Impact**: No central JSON timings format exists to monitor API throughput, endpoint latency, or model/pipeline execution performance.
